## 1. Описание модуля

**Назначение**: Комплекс программ первичной обработки (КППО). Программа вычисления текущих замеров координат (ТЗК). Дискриминаторные оценки координат цели по отсчётам диаграммы направленности антенны (ДН).

**Организация**: ОАО НПК НИИДАР, Москва.

**Авторы кода**:
- Федоров С.А. (2010) — discrcg, discrsd, discrqa
- Добродумов А.Б. (2012) — discrea
- Керский Е.В. (2017) — discr_common, discr_types

**Язык**: C (совместим с C89/C99/C11/C17)

### ERR-001: Функции discr2() и conv3to2() — объявлены, но не реализованы

**Серьёзность**: 🔴 Критическая
**Файл**: `discr_common.h`, строки 37-38, 54-55
**Файл реализации**: отсутствует

**Описание**:
В заголовочном файле `discr_common.h` объявлены два прототипа функций:

```c
// строка 37-38
int discr2( double maz[ 2 ][ 2 ], double mel[ 2 ][ 2 ], double mval[ 2 ][ 2 ],
               double *az, double *el, double *val );

// строка 54-55
int conv3to2( double maz3[ 3 ][ 3 ], double mel3[ 3 ][ 3 ], double mval3[ 3 ][ 3 ],
              double maz2[ 2 ][ 2 ], double mel2[ 2 ][ 2 ], double mval2[ 2 ][ 2 ] );
```

Ни в одном из исходных файлов (.c) реализация этих функций **не найдена**. Поиск выполнен по всем файлам проекта.

**Последствия**:
- При вызове этих функций из внешнего кода произойдёт ошибка линковки (`undefined reference to 'discr2'`).
- Если функции используются в вышестоящем ПО (КППО/ТЗК), то модуль **не может быть успешно слинкован**.
- Функция `conv3to2()` по описанию предназначена для перехода с 3-точечного на 2-точечный дискриминатор при невозможности использования 3-точечного (вогнутость, отсутствие максимума в центре). Её отсутствие означает, что **цепочка обработки «проверка → fallback на 2-точечный» не завершена**.

**Рекомендация**:
Необходимо уточнить у разработчика (Керский Е.В.), были ли эти функции реализованы в другом модуле или проекте. Если нет — требуется реализация для полноты модуля.

### ERR-002: Глобальная статическая переменная — нарушение потокобезопасности

**Серьёзность**: 🔴 Критическая
**Файл**: `discrea.c`, строки 30-34

**Описание**:
В файле `discrea.c` объявлена глобальная статическая структура:

```c
static struct axstr
{
  double a;
  double x;
} ax[ 3 ];
```

Эта переменная используется внутри функции `discr3ea()` (строки 67-73) для сортировки входных данных:

```c
ax[ 0 ].a = A1; ax[ 0 ].x = x1;
ax[ 1 ].a = A2; ax[ 1 ].x = x2;
ax[ 2 ].a = A3; ax[ 2 ].x = x3;
qsort( ax, 3, sizeof( struct axstr ), compare );
```

**Последствия**:
- Переменная `ax[3]` является **разделяемым состоянием** (shared mutable state) между всеми вызовами `discr3ea()`.
- Если функция вызывается **одновременно из нескольких потоков** (например, при параллельной обработке нескольких целей на разных лучах), произойдёт **состояние гонки** (race condition): один поток запишет свои данные в `ax[]`, второй перезапишет их до завершения `qsort()`.
- Результат: **недетерминированное поведение**, неверные оценки координат, потенциально опасная ситуация для системы управления.

**Рекомендация**:
Перенести объявление `ax[3]` внутрь функции `discr3ea()` как локальную переменную:

```c
int discr3ea(double A1, double A2, double A3,
             double x1, double x2, double x3, double* xe)
{
  struct axstr { double a; double x; } ax[3];  // <-- локальная
  // ... остальной код без изменений
}
```

Это не изменяет интерфейс и не влияет на результат вычислений.

### ERR-003: Функция discr3() модифицирует входные данные (побочный эффект)

**Серьёзность**: 🔴 Критическая
**Файл**: `discr_common.c`, строки 74-79

**Описание**:
Функция `discr3()` принимает указатели на массивы координат `maz` и `mel` и **умножает их на коэффициент** `c`:

```c
const double c = 2.0 * M_PI * dx / lambda;
// ...
// строки 74-79:
for ( i = 0; i < ndnx; ++i )
for ( j = 0; j < ndny; ++j )
{
  maz[ i ][ j ] *= c;   // <-- МОДИФИКАЦИЯ ВХОДНЫХ ДАННЫХ
  mel[ i ][ j ] *= c;   // <-- МОДИФИКАЦИЯ ВХОДНЫХ ДАННЫХ
}
```

При этом выходные значения `*az` и `*el` делятся обратно на `c` (строки 108-109):

```c
*az = aztmp / c;
*el = eltmp / c;
```

Однако **входные массивы maz и mel остаются изменёнными** после возврата из функции.

**Последствия**:
1. **Повторный вызов** `discr3()` с теми же массивами даст **неверный результат** — координаты будут умножены на `c` дважды.
2. **Вызывающий код** может использовать `maz`/`mel` после вызова `discr3()` для других целей (отображение, логирование, передача в другие модули) — получит искажённые значения.
3. Если `maz`/`mel` содержат данные, общие для нескольких вызовов (например, сетка координат ДН), то **все последующие обращения к этим данным будут работать с повреждёнными значениями**.

**Рекомендация**:
Работать с локальными копиями входных массивов:

```c
double local_maz[3][3], local_mel[3][3];
for (i = 0; i < ndnx; ++i)
  for (j = 0; j < ndny; ++j) {
    local_maz[i][j] = maz[i][j] * c;
    local_mel[i][j] = mel[i][j] * c;
  }
// далее использовать local_maz, local_mel вместо maz, mel
```

Альтернатива: восстановить входные данные перед возвратом (разделить обратно на `c`). Однако это менее надёжно — при раннем возврате из функции (строка 60: `return -1`) данные останутся изменёнными.

### ERR-004: Использование FLT_EPSILON вместо DBL_EPSILON для переменных типа double

**Серьёзность**: 🟡 Важная
**Файлы**: `discrqa.c` строки 45, 47; `discr_common.c` строка 130

**Описание**:
В нескольких местах для сравнения значений типа `double` используется константа `FLT_EPSILON` (порог для типа `float`, ~1.19e-7), вместо `DBL_EPSILON` (порог для типа `double`, ~2.22e-16):

**discrqa.c, строка 45**:
```c
if (fabs(A2-A3) < FLT_EPSILON)  // FLT_EPSILON ≈ 1.19e-7
```

**discrqa.c, строка 47**:
```c
if (fabs(A2-A1) < FLT_EPSILON)
```

**discr_common.c, строка 130**:
```c
if ( mval[ i ][ j ] < FLT_EPSILON )
```

При этом в файле `discrea.c` **корректно** используется `DBL_EPSILON`:
```c
if (A1 < DBL_EPSILON || A2 < DBL_EPSILON || A3 < DBL_EPSILON)  // строка 57
```

**Последствия**:
- `FLT_EPSILON` (~1e-7) в 10⁹ раз грубее чем `DBL_EPSILON` (~2e-16).
- Для `discrqa.c`: амплитуды, отличающиеся менее чем на 1e-7 но более чем на 2e-16, будут ошибочно приняты за равные. Это приведёт к выбору неверной ветки алгоритма (например, `return x2` вместо вычисления по формуле).
- Для `checkuse3d`: амплитуды в диапазоне (2e-16, 1e-7) будут ошибочно считаться нулевыми, и функция вернёт 0 (невозможно использовать 3-точечный дискриминатор), хотя данные корректны.

**Рекомендация**:
Заменить `FLT_EPSILON` на `DBL_EPSILON` во всех случаях сравнения `double` значений. Или ввести явную константу порога (например, `#define DISCR_EPS 1e-12`), чтобы контролировать чувствительность независимо от типа данных.

### ERR-005: Отсутствие защиты от деления на ноль

**Серьёзность**: 🟡 Важная
**Файлы**: `discrcg.c` строки 42, 64, 88; `discrsd.c` строка 48

**Описание**:
Функции центра тяжести (CG) и суммарно-разностного (SD) дискриминаторов выполняют деление на сумму амплитуд без проверки на ноль:

**discr2cg (строка 42)**:
```c
xe = (A1*x1 + A2*x2) / (A1 + A2);  // если A1 = -A2 или A1 = A2 = 0 → деление на 0
```

**discr3cg (строка 64)**:
```c
xe = (A1*x1 + A2*x2 + A3*x3) / (A1 + A2 + A3);  // аналогично
```

**discrncg (строка 88)**:
```c
xe = Axs / As;  // As = sum(A[i]), может быть 0
```

**discr2sd (строка 48)**:
```c
dx = c * ((A2 - A1) / (A2 + A1));  // если A1 = A2 = 0 → деление на 0
```

**Последствия**:
- При нулевых (или компенсирующих друг друга) амплитудах результат будет `NaN` (Not a Number) или `Inf` (бесконечность).
- `NaN` будет распространяться дальше по цепочке обработки: через `discr3()`, далее в вышестоящее ПО (ТЗК/КППО).
- Результат: **некорректная оценка координат цели**, потенциально опасная для системы управления.

**Примечание**: В файлах `discrqa.c` и `discrea.c` аналогичные проверки **есть** — эти функции корректно обрабатывают вырожденные случаи. Отсутствие проверок в CG и SD — **непоследовательность** кода.

**Рекомендация**:
Добавить проверки перед делением:

```c
double discr2cg(double A1, double A2, double x1, double x2)
{
  double sum = A1 + A2;
  if (fabs(sum) < DBL_EPSILON)
    return (x1 + x2) * 0.5;  // при нулевых амплитудах — середина отрезка
  return (A1 * x1 + A2 * x2) / sum;
}
```

Для `discr2sd` аналогично — при A1+A2=0 вернуть среднее значение координат.

### ERR-006: Функция discr3() не поддерживает типы DT_CG и DT_SD

**Серьёзность**: 🟢 Замечание (архитектура)
**Файл**: `discr_common.c`, строки 81-106

**Описание**:
Функция-обёртка `discr3()` принимает параметр `DISCRTYPE discrtype`, но в `switch` обрабатывает только два типа из пяти:

```c
switch ( discrtype )
{
  case DT_QA:   // ✅ обрабатывается
    // ...
    break;
  case DT_EA:   // ✅ обрабатывается
    // ...
    break;
  default:      // DT_NO, DT_CG, DT_SD → ошибка -2
    *az = 0.0; *el = 0.0; *val = 0.0;
    return -2;
}
```

**Последствия**:
- Вызов `discr3(maz, mel, mval, 3, 3, DT_CG, dx, lambda, &az, &el, &val)` вернёт -2 и обнулит координаты, хотя CG-дискриминатор реализован в `discrcg.c` и **мог бы быть вызван**.
- Аналогично для DT_SD — функция `discr2sd()` существует, но не доступна через обёртку.
- Это ограничивает универсальность API: вызывающий код не может использовать единую точку входа для всех типов дискриминаторов.

**Рекомендация**:
Добавить обработку DT_CG и DT_SD в `switch`, используя существующие функции `discr3cg()` и `discr2sd()`:

```c
case DT_CG:
  aztmp = discr3cg(mval[maxRow][0], mval[maxRow][1], mval[maxRow][2],
                   maz[maxRow][0],  maz[maxRow][1],  maz[maxRow][2]);
  eltmp = discr3cg(mval[0][maxCol], mval[1][maxCol], mval[2][maxCol],
                   mel[0][maxCol],  mel[1][maxCol],  mel[2][maxCol]);
  *val = maxAmpl;
  break;
```

### STYLE-001: Смешанные отступы (табуляция + пробелы)

**Серьёзность**: 🟢 Стиль
**Файл**: `discrqa.c`, строки 47-59

**Описание**:
В файле `discrqa.c` используются **табуляции** для отступов (строки 47-59), в то время как остальные файлы проекта используют **пробелы**. Это приводит к неоднородному отображению кода в разных редакторах и IDE.

```c
if (fabs(A2-A3)<FLT_EPSILON)
{
⇥ if(fabs(A2-A1)<FLT_EPSILON)    // ← табуляция
⇥ ⇥ return x2;                    // ← двойная табуляция
⇥ if(A1>A2)
⇥ ⇥ return x1;
```

**Рекомендация**: Заменить все табуляции на 2 пробела (стандарт Google C++ Style Guide, принятый в проекте).

**Серьёзность**: 🟢 Стиль
**Файлы**: `discr_common.c`, `discrea.c`

**Описание**:

int discr3ea(...) {

  double a1, a2, a3, a12, a23, a13;

**Рекомендация**: При рефакторинге на C17 переменные можно объявлять ближе к месту первого использования. Это улучшает читаемость, но НЕ является обязательным исправлением.
